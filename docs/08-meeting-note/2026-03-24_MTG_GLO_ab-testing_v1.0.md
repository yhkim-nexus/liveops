# 회의록: A/B 테스트(실험) 기능 기획

> Game LiveOps Service의 Phase 2 핵심 기능인 A/B 테스트 기능을 기획하는 세션. 경쟁사 분석 결과를 바탕으로 설계 방식(3가지 옵션), 주요 사용자, 실험 대상, 통계 수준, 세그먼트 연동, 사전 테스트, 승인 워크플로우를 논의하고 원격 설정 확장형 설계 방식을 확정한다.

## 1. 회의 목적

Game LiveOps Service의 Phase 2에 해당하는 A/B 테스트(실험) 기능을 기획하여 MVP 설계 방식을 확정한다. 경쟁사 분석 결과(RES-GLO-002 Section 3.3)를 바탕으로 비기술 운영팀을 주요 사용자로 하는 실험 생성·설정·승인·분석 워크플로우를 정의하고, 기존 세그멘테이션 아키텍처(PRD-GLO-001)와 라이브 이벤트(PRD-GLO-002)와의 통합 방식을 확보한다.

---

## 2. 논의 내용

### 2.1 A/B 테스트의 위치와 중요도

**배경**
- RES-GLO-002의 경쟁사 분석에서 A/B 테스트는 7개 플랫폼(Satori, Unity, AccelByte, Hive, Balancy, GameAnalytics, Metaplay) 모두 지원하는 Must-Have #4 기능
- 변형(Variant) 정의와 오디언스 타겟 롤아웃은 전 플랫폼 공통 기능
- 목표 지표 추적(5/7), 비공개 사전 테스트(3/7), 통계적 유의성 판단(3/7)은 차별화 요소

**논의 내용**
- A/B 테스트는 세그멘테이션(P0) 및 라이브 이벤트(P1) 다음으로 높은 우선순위를 가지는 기능
- 경쟁사 벤치마킹: Heroic Labs Satori의 "원격 설정 기반 실험", Unity의 "Game Overrides 모델", AccelByte의 "A/B 테스트 기본 틀"
- 운영팀이 코드 없이 실험을 생성하고, 데이터 기반 의사결정을 지원하는 것이 핵심

**결론**
- A/B 테스트를 GLO 서비스 Phase 2의 핵심 기능(P1)으로 확정
- 비기술 LiveOps 운영팀을 주요 사용자로 설정
- 5단계 위저드 기반 직관적 UI 제공 필수

### 2.2 설계 방식 3가지 옵션 검토

**배경**
- A/B 테스트 기능을 구현하는 방식에는 3가지 주요 패턴 존재
- 각 방식의 장단점을 분석하여 GLO 서비스에 최적인 방식 선택 필요

**논의 내용**

#### 2.2.1 옵션 A: 원격 설정 확장형 (Remote Config Extension)

| 항목 | 설명 |
|------|------|
| **개념** | A/B 테스트를 "조건부 원격 설정 + 트래픽 분배 + 지표 추적"으로 정의 |
| **구현** | 원격 설정 시스템에 트래픽 분배(%), 통계 분석(p-value, CI) 기능 확장 |
| **참고 모델** | Heroic Labs Satori 2.1+, Unity Game Overrides |
| **장점** | 실험 = 조건부 원격 설정으로 매우 직관적, 기존 원격 설정 키 재사용 가능, 학습 곡선 짧음 |
| **단점** | 원격 설정 시스템이 먼저 구축되어야 함 (의존성) |
| **추천도** | ★★★★★ (적극 권장) |

#### 2.2.2 옵션 B: 독립 실험 모듈형 (Standalone Experiment Module)

| 항목 | 설명 |
|------|------|
| **개념** | A/B 테스트를 독립적인 실험 시스템으로 설계 |
| **구현** | 별도 EXPERIMENT 테이블, 변형 관리, 트래픽 분배, 분석 파이프라인 구축 |
| **참고 모델** | Optimizely, VWO 등 A/B 테스트 전문 SaaS |
| **장점** | 원격 설정 없이도 독립적으로 동작 가능, 복잡한 실험 설계 지원 용이 |
| **단점** | 원격 설정과의 데이터 모델 불일치, 관리 복잡도 증가, 이벤트 변형 연동 어려움 |
| **추천도** | ★★☆☆☆ (비추천) |

#### 2.2.3 옵션 C: 이벤트 내장형 (Event-Embedded)

| 항목 | 설명 |
|------|------|
| **개념** | A/B 테스트를 라이브 이벤트(PRD-GLO-002)의 기능으로 내장 |
| **구현** | 이벤트 생성 시 "실험 모드" 옵션으로 다중 변형 정의, 종료 후 승자 자동 선택 |
| **참고 모델** | Metaplay의 AB Testing 레이어 |
| **장점** | 라이브 이벤트와 완전 통합, 이벤트 승인 워크플로우 재사용 |
| **단점** | 원격 설정이나 Feature Flag 변형 실험 불가능, 확장성 제약 |
| **추천도** | ★★☆☆☆ (비추천, 순수 이벤트 A/B만 가능) |

**결론**
- **옵션 A(원격 설정 확장형)를 기본 설계 방식으로 채택** (D-019)
- Satori/Unity 모델을 참고하여 "실험 = 조건부 원격 설정 + 트래픽 분배 + 통계 분석"으로 정의
- 원격 설정 기능과의 강한 통합으로 관리 일관성 확보

### 2.3 주요 사용자 및 UI 전략

**배경**
- A/B 테스트 기능의 접근성과 채택률은 사용자 친화성에 달려있음
- Satori는 기술자/비기술자 모두 지원, Unity는 비기술자 중심, Optimizely는 데이터 분석가 중심

**논의 내용**
- 주요 사용자는 **비기술 LiveOps 운영팀** (게임 운영 담당자, 커뮤니티 매니저 등)
- 운영팀이 코드 없이 실험을 생성할 수 있는 **5단계 위저드 UI** 제공 필수
  - Step 1: 기본 정보 (실험명, 설명, 가설)
  - Step 2: 변형 정의 (컨트롤 + 최대 3개 변형, 트래픽 %)
  - Step 3: 오디언스 선택 (기존 오디언스 드롭다운)
  - Step 4: 목표 지표 설정 (Primary 1개 + Secondary 최대 3개)
  - Step 5: 확인 및 저장

**결론**
- 주요 사용자를 **비기술 LiveOps 운영팀**으로 명확히 정의 (D-020)
- **5단계 위저드 기반 폼 인터페이스** MVP 필수 포함
- 고급 사용자용 쿼리 모드는 Phase 2 이후 고려

### 2.4 실험 대상(Variant) 유형 및 범위

**배경**
- A/B 테스트가 원격 설정 확장형이므로, 실험 대상은 원격 설정뿐 아니라 다양한 변형 가능
- Satori는 "원격 설정 + Feature Flag" 복합 변형, Unity는 주로 "Game Overrides(원격 설정)", AccelByte는 "스킨, 이벤트, 보상" 복합

**논의 내용**

| 변형 유형 | 내용 | 예시 |
|---------|------|------|
| 원격 설정 변형 | Key-Value 파라미터 변경 | 난이도 easy vs hard, 보상량 100 vs 200 |
| 이벤트 변형 | 이벤트 구성 변경 (PRD-GLO-002와 연동) | 이벤트 기간 3일 vs 7일, 보상 종류 A vs B |
| Feature Flag 변형 | 기능 활성화/비활성화 | 신규 UI on vs off, 베타 기능 활성화 |
| 복합 변형 | 위 항목의 조합 | 난이도 + 보상 + UI 동시 변경 |

- MVP 범위: 원격 설정 + 이벤트 + Feature Flag + 조합까지 모두 지원 (분산 레이어 없이 독립 할당)
- 동일 오디언스에 복수 실험 실행 가능하나, 실험 생성 시 충돌 경고 표시 (차단 X)

**결론**
- 실험 대상을 **복합 (원격 설정 + 이벤트 + Feature Flag + 조합)** 으로 정의 (D-021)
- MVP는 **독립 할당 방식** (각 실험이 독립적으로 해시 기반 할당)
- 멀티 레이어(상호 배타적 격리)는 Phase 2 이후로 미룸

### 2.5 통계 수준 및 분석 기능

**배경**
- 경쟁사별 통계 역량 차이 큼: 기본(빈도 기반) → 중급(p-value, CI) → 고급(베이지안, 자동 최적화)
- 게임 라이브옵스의 경우 의사결정 속도 중요하므로, 중급 통계로도 충분하지만 신뢰성 있는 승자 판별 필수

**논의 내용**

| 분석 항목 | MVP 포함 | 설명 |
|----------|---------|------|
| **검정 방법** | ✓ | Chi-squared (전환율), Welch's t-test (평균값/횟수) |
| **p-value** | ✓ | 통계적 유의성 판단 (α=0.05 기본) |
| **신뢰구간(CI)** | ✓ | 95% CI 표시 |
| **Bonferroni 보정** | ✓ | 다중 비교 보정 (변형 2개 이상 시 적용) |
| **승자 자동 판별** | ✓ | p < 조정된 α이고 최고 성과 변형을 "승자"로 표시 |
| **MDE 계산** | ✓ | 최소 감지 효과(%) 입력 시 필요 샘플 사이즈 자동 계산 |
| **베이지안 통계** | ✗ | Phase 2 이후 |
| **자동 중단** | ✗ | Phase 2 이후 |
| **MAB 최적화** | ✗ | Phase 2 이후 |

- **검정 수식**: Chi-squared test (전환율), Welch's t-test (연속형), Bonferroni 조정
- **승자 판별 로직**: p < 조정된 α (예: 컨트롤 + 3개 변형 시 α/3 ≈ 0.0167) AND primary metric 최고 성과

**결론**
- 통계 수준을 **중급 (Chi-squared, Welch's t-test, Bonferroni 보정, 승자 자동 판별)** 으로 정의 (D-022)
- MVP에서 신뢰성 있는 A/B 테스트 의사결정 지원
- 고급 통계(베이지안, 자동 중단, MAB)는 Phase 2 이후

### 2.6 세그먼트 연동 및 오디언스 활용

**배경**
- Phase 1에서 설계한 세그멘테이션 아키텍처(PRD-GLO-001 F-006)가 라이브 이벤트(PRD-GLO-002)와 완전 통합되었음
- A/B 테스트도 동일한 오디언스 시스템을 활용하여 중복 로직 제거 필요

**논의 내용**
- A/B 테스트의 타겟 오디언스는 **PRD-GLO-001의 오디언스(Audience) 엔티티를 직접 재사용**
- 실험 생성 시 Step 3에서 기존 오디언스 드롭다운 선택 또는 "전체 플레이어" 선택
- 실험 타겟 변경은 불가능 (running 상태로 진입 후) → sticky 할당으로 무결성 보장
- 오디언스가 삭제/비활성화되면 실험은 paused 상태로 자동 전환

**결론**
- 세그먼트 연동을 **기존 오디언스(PRD-GLO-001 F-006) 직접 활용** 으로 정의 (D-023)
- 오디언스 시스템의 재사용으로 관리 일관성 및 복잡도 감소

### 2.7 사전 테스트(Testing) 상태 및 QA 프로세스

**배경**
- 실험이 오배포되어 실제 플레이어에게 노출되면 서비스 신뢰도 저하
- Satori는 "Inactive" 상태로 내부 테스트 후 활성화, Metaplay도 "Testing" 상태 사전 제공

**논의 내용**
- 실험이 `draft` → `testing` → `pending_approval` → `running` 상태로 진행
- `testing` 상태에서는 **내부 QA 그룹(예: 개발 팀 일부, 테스터 세그먼트)에만 노출**
- 실제 플레이어는 testing 상태에서 노출 받지 않음 (관리자 권한 필요)
- QA 확인 완료 후 `testing` → `pending_approval` 전환 (승인 요청)

**결론**
- 사전 테스트(testing 상태)를 **필수로 포함** (D-024)
- 라이브 배포 전 내부 QA 완료 프로세스 보장

### 2.8 승인 워크플로우(Approval Workflow) 설계

**배경**
- 라이브 이벤트(PRD-GLO-002)에서 검증된 2단계 승인 워크플로우 패턴 재사용
- 기획자/운영팀(요청) → 승인자(검토 및 승인) 역할 분리로 운영 안전성 확보

**논의 내용**

| 단계 | 역할 | 액션 | 결과 |
|------|------|------|------|
| 1 | 기획자/운영팀 | testing 상태의 실험을 pending_approval로 전환 (승인 요청) | pending_approval |
| 2 | 승인자 | 실험 설정 검토 (정책 위반, 오디언스 유효성, 중복 확인) | - |
| 3 | 승인자 | 승인 또는 반려 | approved or draft (반려 시 재편집) |
| 4 (승인 시) | 시스템 | running 상태로 자동 전환 (또는 일정 시간 후 자동 전환) | running |

- **자가 승인 금지**: 실험 생성자와 승인자는 동일 인물일 수 없음
- **승인 검토 체크리스트** (UI에 제시):
  - 실험명 중복 여부
  - 타겟 오디언스 멤버 수 (0인 경우 경고)
  - 오디언스 내 활성 실험 충돌 표시
  - Primary 지표 선택 여부
  - 실험 기간 합리성 (MDE 기반 최소 기간)

**결론**
- 승인 워크플로우를 **필수로 포함** (D-025)
- PRD-GLO-002의 이벤트 승인 패턴과 동일한 체계 적용

### 2.9 실험 상태 머신 및 전체 생명 주기

**배경**
- SPEC-GLO-001에서 정의한 8단계 상태 머신 검토: draft → testing → pending_approval → running → analyzing → completed (paused, stopped 중간 상태 포함)

**논의 내용**
- 상태별 가능한 액션 및 전이 규칙은 SPEC-GLO-001 Section 4 참조
- MVP 범위: draft → testing → pending_approval → running → analyzing → completed (중간 상태: paused, stopped 포함)

**결론**
- 상태 머신 및 상태별 권한/액션은 **SPEC-GLO-001 정의 기준 준수** (기획 단계에서 별도 확정 불필요)

## 3. 결정 사항
