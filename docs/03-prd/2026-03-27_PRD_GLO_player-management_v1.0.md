# PRD: 플레이어 상태 조회 및 계정 조치

> 운영팀이 개별 플레이어를 검색·조회하고 계정 차단/해제/닉네임 변경 등 운영 조치를 수행할 수 있는 플레이어 관리 기능을 정의한다.

## 1. Overview

### 1.1 제품 비전

운영팀이 개별 플레이어를 통합 검색바로 빠르게 찾고, 프로필 상세에서 상태·활동·소속 오디언스를 한눈에 파악하며, 필요 시 계정 차단/해제/닉네임 변경 등 즉각적인 운영 조치를 수행할 수 있는 플레이어 관리 시스템을 구축한다.

### 1.2 배경 및 목적

**배경**

- 경쟁사 분석(RES-GLO-002)에서 "플레이어 상태 조회"는 Must-Have #7로 분류되었다. 5/7 경쟁 플랫폼이 플레이어 관리 기능을 지원한다.
- "계정 조치(차단/해제)" 기능은 4/7 경쟁 플랫폼이 지원하며, CS 운영의 핵심 기능으로 분류된다.
- Metaplay의 플레이어 프로필 통합 뷰와 실시간 검색, 계정 조치 이력 추적 모델을 벤치마킹한다.
- PRD-GLO-005(RBAC)에서 정의된 Operator 역할이 계정 조치 권한을 보유하며, 감사 로그와 연동하여 모든 조치를 기록한다.

**목적**

- 이 PRD는 GLO 서비스의 **플레이어 상태 조회 및 계정 조치** 기능 범위를 정의한다.
- 플레이어 검색(통합 검색, 디바운스, 페이지네이션), 프로필 상세(기본 정보, 소속 오디언스, 계정 상태), 계정 조치(차단/해제/닉네임 변경)를 명세한다.
- MVP(Phase 1) 출시에 필요한 최소 기능과 수용 기준을 확정한다.

### 1.3 성공 지표 (Success Metrics)

| 지표 | 현재 | 목표 | 측정 방법 |
|------|------|------|----------|
| 플레이어 검색 응답 시간 | - | 500ms 이내 | 검색어 입력 후 결과 테이블 렌더링 완료까지 측정 |
| 검색 정확도 | - | 100% | ID/닉네임/이메일 부분 일치 검색에서 해당 플레이어가 결과에 포함됨 |
| 계정 조치 성공률 | - | 100% | 차단/해제/닉네임 변경 요청이 정상적으로 반영됨 |
| 조치 이력 기록률 | - | 100% | 모든 계정 조치가 감사 로그에 기록됨 |
| 권한 검증 정확도 | - | 100% | Viewer 역할로 계정 조치 시도 시 100% 차단 |

## 2. Target Users

### 2.1 사용자 페르소나

#### Primary Persona: CS 운영자 (Operator)

| 항목 | 내용 |
|------|------|
| 소속 | 게임사 CS팀 / LiveOps 운영팀 |
| 역할 | 플레이어 문의 처리, 계정 관리, 운영 조치 |
| 기술 수준 | 비개발자. 관리 도구 사용에 익숙 |
| 주요 도구 | 관리자 대시보드, CS 티켓 시스템 |

**Goals**
- 플레이어 ID나 닉네임으로 빠르게 해당 플레이어를 찾고 싶다.
- 한 화면에서 플레이어의 상태, 활동, 소속 오디언스를 모두 확인하고 싶다.
- 부적절한 행위자를 즉시 차단하고, 오차단된 플레이어를 빠르게 해제하고 싶다.

**Pain Points**
- 플레이어를 찾기 위해 여러 시스템을 전환해야 한다.
- 계정 조치 시 사유가 기록되지 않아 추후 추적이 어렵다.
- 차단 해제 시 어떤 사유로 차단되었는지 확인이 불가능하다.

#### Secondary Persona: 팀 리드 (Admin)

| 항목 | 내용 |
|------|------|
| 소속 | 게임사 운영팀 리드 |
| 역할 | CS 운영 감독, 조치 이력 모니터링, 에스컬레이션 처리 |
| 기술 수준 | 비개발자. 데이터 분석 기초 이해 |

**Goals**
- CS 운영자들의 계정 조치 이력을 모니터링하고 싶다.
- 부적절한 계정 조치를 빠르게 발견하고 교정하고 싶다.

## 3. Features & Requirements

### 3.1 F-034: 플레이어 검색

**설명:** 통합 검색바에서 플레이어 ID, 닉네임, 이메일을 동시에 검색하여 결과 테이블에 표시한다.

#### 유저 스토리

| ID | 스토리 | 우선순위 |
|----|--------|---------|
| US-049 | CS 운영자로서, 플레이어 ID나 닉네임을 입력하면 해당 플레이어를 빠르게 찾을 수 있다 | P0 |
| US-050 | CS 운영자로서, 검색 결과에서 플레이어의 상태와 기본 정보를 한눈에 파악할 수 있다 | P0 |

#### 요구사항

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| REQ-034-01 | 단일 검색바에서 플레이어 ID, 닉네임, 이메일을 부분 일치 방식으로 동시에 검색한다 | P0 |
| REQ-034-02 | 검색 입력에 300ms 디바운스를 적용하여 불필요한 API 호출을 방지한다 | P0 |
| REQ-034-03 | 검색 결과를 테이블 형태로 표시한다: 닉네임, 이메일, 레벨, 상태, 플랫폼, 누적 결제, 최근 로그인 | P0 |
| REQ-034-04 | 플레이어 상태를 색상 뱃지로 시각적으로 구분한다: 활성(녹색), 차단(적색), 정지(주황) | P0 |
| REQ-034-05 | 페이지당 10건씩 페이지네이션을 제공한다 | P0 |
| REQ-034-06 | 검색어가 비어 있으면 전체 플레이어 목록을 표시한다 | P1 |
| REQ-034-07 | 검색 결과의 플레이어 행을 클릭하면 해당 플레이어 상세 페이지(`/players/[id]`)로 이동한다 | P0 |

#### 수용 기준 (Acceptance Criteria)

**AC-034-01: 통합 검색 실행**

```gherkin
Scenario: 검색어 입력 시 플레이어 검색 결과 표시
  Given 검색바가 활성화된 상태이다
  When "alpha"를 입력하고 300ms가 경과한다
  Then 닉네임 또는 이메일에 "alpha"가 포함된 플레이어가 테이블에 표시된다
```

**AC-034-02: 페이지네이션 동작**

```gherkin
Scenario: 검색 결과가 10건 초과 시 페이지네이션
  Given 검색 결과가 11건 이상이다
  When 2페이지 버튼을 클릭한다
  Then 11번째 플레이어부터 결과가 표시된다
```

**AC-034-03: 검색 결과 없음**

```gherkin
Scenario: 검색 결과가 없을 때 빈 상태 표시
  Given 검색바가 활성화된 상태이다
  When 존재하지 않는 검색어를 입력하고 300ms가 경과한다
  Then "검색 결과가 없습니다" 빈 상태 메시지가 표시된다
```

```gherkin
Scenario: 차단된 플레이어의 차단 정보 표시
  Given 플레이어의 status가 "banned"이다
  When 해당 플레이어의 상세 페이지에 접근한다
  Then 차단 사유(banReason)와 차단 일시(bannedAt)가 표시된다
```

### 3.3 F-036: 계정 조치

**설명:** Operator 이상 권한을 가진 관리자가 플레이어 상세 페이지에서 계정 차단, 해제, 닉네임 변경을 수행한다.

#### 유저 스토리

| ID | 스토리 | 우선순위 |
|----|--------|---------|
| US-053 | CS 운영자로서, 부적절한 행위를 한 플레이어를 사유를 명시하여 차단할 수 있다 | P0 |
| US-054 | CS 운영자로서, 오차단된 플레이어의 차단을 해제할 수 있다 | P0 |
| US-055 | CS 운영자로서, 부적절한 닉네임을 가진 플레이어의 닉네임을 변경할 수 있다 | P0 |

#### 요구사항

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| REQ-036-01 | 차단 조치 시 다이얼로그에서 차단 사유를 입력받는다. 최소 10자 이상 필수 | P0 |
| REQ-036-02 | 차단 실행 시 플레이어 상태를 `banned`로 변경하고, `banReason`과 `bannedAt`을 기록한다 | P0 |
| REQ-036-03 | 차단 해제 시 플레이어 상태를 `active`로 변경하고, `banReason`과 `bannedAt`을 초기화한다 | P0 |
| REQ-036-04 | 닉네임 변경 시 다이얼로그에서 새 닉네임을 입력받는다 | P0 |
| REQ-036-05 | 모든 계정 조치는 Operator 이상 역할(RBAC)이 필요하다. Viewer 역할은 조치 버튼이 비활성화된다 | P0 |
| REQ-036-06 | 이미 차단된 플레이어에게는 "차단" 버튼 대신 "차단 해제" 버튼을 표시한다 | P0 |
| REQ-036-07 | 모든 계정 조치는 감사 로그에 기록한다: 조치 유형, 대상 플레이어, 실행 관리자, 일시 | P1 |
| REQ-036-08 | 차단 사유가 10자 미만일 때 차단 버튼을 비활성화하고, 유효성 검사 메시지를 표시한다 | P0 |
| REQ-036-09 | active 상태 플레이어에 대해 임시 정지(suspend) 조치를 수행할 수 있다. 사유(10자 이상)와 기간(1/3/7/30일)을 입력한다 | P0 |
| REQ-036-10 | suspended 상태 플레이어를 수동으로 정지 해제(unsuspend)할 수 있다 | P0 |
| REQ-036-11 | suspended 상태 플레이어에 대해 영구 차단(ban) 조치로 전환할 수 있다 | P0 |
| REQ-036-12 | 임시 정지 시 suspendReason, suspendedAt, suspendedUntil이 기록된다 | P0 |

#### 수용 기준 (Acceptance Criteria)

**AC-036-01: 플레이어 차단**

```gherkin
Scenario: 활성 플레이어를 차단한다
  Given 플레이어 상태가 "active"이고 관리자 역할이 Operator 이상이다
  When "차단" 버튼을 클릭하고 10자 이상의 차단 사유를 입력한 후 확인한다
  Then 상태가 "차단됨"으로 변경되고 차단 사유와 차단 일시가 표시된다
```

**AC-036-02: 차단 해제**

```gherkin
Scenario: 차단된 플레이어를 해제한다
  Given 플레이어 상태가 "banned"이고 관리자 역할이 Operator 이상이다
  When "차단 해제" 버튼을 클릭한다
  Then 상태가 "활성"으로 변경되고 차단 정보가 초기화된다
```

**AC-036-03: 닉네임 변경**

```gherkin
Scenario: 플레이어 닉네임을 변경한다
  Given 플레이어 상세 페이지에 접근했고 관리자 역할이 Operator 이상이다
  When "닉네임 변경" 버튼을 클릭하고 새 닉네임을 입력한 후 확인한다
  Then 닉네임이 변경되어 화면에 반영된다
```

**AC-036-04: 권한 없는 사용자 접근 제한**

```gherkin
Scenario: Viewer 역할은 계정 조치를 수행할 수 없다
  Given 관리자 역할이 Viewer이다
  When 플레이어 상세 페이지에 접근한다
  Then 차단, 차단 해제, 닉네임 변경 버튼이 비활성화 또는 미표시된다
```

## 5. Data Model

### 5.1 Player 엔티티

`apps/admin/src/features/players/types/player.ts` 기반 데이터 모델:

```typescript
type PlayerStatus = "active" | "banned" | "suspended";
type Platform = "iOS" | "Android" | "PC";

interface Player {
  id: string;           // 플레이어 고유 ID (예: "p-10001")
  nickname: string;     // 닉네임
  email: string;        // 이메일
  level: number;        // 레벨
  status: PlayerStatus; // 계정 상태
  platform: Platform;   // 플랫폼
  country: string;      // 국가 코드 (ISO 3166-1 alpha-2)
  totalSpent: number;   // 누적 결제액 (USD)
  lastLoginAt: string;  // 최근 로그인 일시 (ISO 8601)
  createdAt: string;    // 가입일 (ISO 8601)
  banReason?: string;   // 차단 사유 (차단 시에만)
  bannedAt?: string;    // 차단 일시 (차단 시에만)
  suspendReason?: string;   // 정지 사유 (정지 시에만)
  suspendedAt?: string;     // 정지 일시 (ISO 8601, 정지 시에만)
  suspendedUntil?: string;  // 정지 만료일 (ISO 8601, 정지 시에만)
  lastDeviceModel: string; // 마지막 접속 기기 모델명 (예: "iPhone 14 Pro", "Samsung Galaxy S23", "PC-Windows")
  lastDeviceOS: string;    // 마지막 접속 OS (예: "iOS 17.4", "Android 14", "Windows 11")
  logicVersion: number;    // 플레이어의 마지막 접속 Logic Version (예: 15, 16, 17)
  audiences: string[];  // 소속 오디언스 이름 목록 (비정규화된 이름 배열, 아래 참고)
}
```

> **audiences 필드 설명**: 현재 Mock 구현(`apps/admin/src/features/players/types/player.ts`)에서는 오디언스 이름을 `string[]` 배열로 비정규화하여 저장한다. 이는 플레이어 목록/상세 화면에서 소속 오디언스를 즉시 표시하기 위한 설계이다. 실제 프로덕션 환경에서는 AUDIENCE 엔티티와 M:N 관계(정규화 모델)로 전환하여 데이터 정합성을 보장할 예정이다. 세그먼테이션 기능(PRD-GLO-001)의 오디언스 데이터와 연동된다.

## 6. API Endpoints

| # | Method | Endpoint | 설명 | 권한 |
|---|--------|----------|------|------|
| 1 | GET | `/api/players?q={query}&page={n}&size={n}` | 플레이어 검색 및 목록 조회 | Viewer+ |
| 2 | GET | `/api/players/{id}` | 플레이어 상세 조회 | Viewer+ |
| 3 | POST | `/api/players/{id}/ban` | 플레이어 차단 (`{ reason: string }`) | Operator+ |
| 4 | POST | `/api/players/{id}/unban` | 플레이어 차단 해제 | Operator+ |
| 5 | PATCH | `/api/players/{id}/nickname` | 닉네임 변경 (`{ nickname: string }`) | Operator+ |
| 6 | POST | `/api/players/{id}/suspend` | 플레이어 임시 정지 (`{ reason: string, durationDays: number }`) | Operator+ |
| 7 | POST | `/api/players/{id}/unsuspend` | 플레이어 정지 해제 | Operator+ |

## 7. Dependencies & Constraints

### 7.1 선행 의존성

| 의존 대상 | 문서 ID | 설명 |
|-----------|---------|------|
| RBAC 시스템 | PRD-GLO-005 (F-029) | 역할 기반 접근 제어. Operator 이상 역할이 계정 조치 권한을 보유해야 한다 |
| 세그먼테이션 | PRD-GLO-001 | 오디언스 멤버십 데이터. 플레이어 상세의 소속 오디언스 목록은 세그먼테이션 기능에서 관리하는 오디언스 데이터에 의존한다 |

### 7.2 제약 사항

| 제약 | 설명 |
|------|------|
| Mock 데이터 전용 | 현재 구현은 Mock API 기반이며 실제 플레이어 데이터베이스와 연동하지 않는다. 실제 백엔드 연동 시 API 스펙 변경이 필요할 수 있다 |
| RBAC Operator 역할 전제 | 계정 조치(차단/해제/닉네임 변경) 기능은 PRD-GLO-005의 RBAC Operator 역할이 구현되어 있어야 정상 동작한다 |
| 클라이언트 사이드 검색 | 현재 Mock 환경에서는 클라이언트 사이드 필터링으로 검색을 수행하며, 대규모 데이터 시 서버 사이드 검색으로 전환이 필요하다 |
