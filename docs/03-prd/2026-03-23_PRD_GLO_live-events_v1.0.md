## 1. Overview

### 1.1 제품 비전

인디~중소 게임사 운영자가 코드 작성 없이 라이브 이벤트를 생성·스케줄링·승인·모니터링할 수 있는 통합 이벤트 관리 시스템을 제공한다. 기획자, 승인 권한자, 개발자 모두가 단일 인터페이스에서 이벤트 전 주기(생성 → 승인 → 활성화 → 종료 → 분석)를 운영한다.

### 1.2 배경 및 목적

**배경**

- 경쟁사 분석(RES-GLO-002, Section 3.2)에서 라이브 이벤트 관리는 7개 플랫폼 중 6개가 지원하는 Must-Have #3 기능이다. 이벤트 생성 및 스케줄링, 세그먼트 기반 타겟팅, 실시간 성과 데이터가 공통 요소다. (RES-GLO-002, Section 4.1)
- PLO Phase 1(세그멘테이션, PRD-GLO-001)에서 구축한 AUDIENCE 테이블과 오디언스 타겟팅 인프라(F-006)를 Live Events가 직접 활용한다.
- 이벤트 오배포·실수로 인한 서비스 운영 사고를 방지하기 위해 승인 워크플로우로 운영 안전성을 확보한다.
- Heroic Labs Satori의 이벤트 캘린더(1/2/4주 뷰), AccelByte의 시즌 패스 구조, Metaplay의 상태 기반 이벤트 관리 방식을 벤치마킹한다.

**목적**

- 이 PRD는 GLO 서비스의 **라이브 이벤트 생성 및 스케줄링** 기능 범위를 정의한다.
- 이벤트 CRUD, 스케줄링(단순/반복), 승인 워크플로우, 오디언스 타겟팅 연동, 상태 머신, 캘린더 뷰, 성과 대시보드, 웹훅/알림을 명세한다.
- MVP(Phase 1) 출시에 필요한 최소 기능과 수용 기준을 확정한다.

### 1.3 성공 지표 (Success Metrics)

| 지표 | 현재 | 목표 | 측정 방법 |

## 2. Target Users

### 2.1 사용자 페르소나

#### Primary Persona: 김지연 (게임 기획자/LiveOps 운영자)

PRD-GLO-001에서 정의된 동일 페르소나를 재사용한다.

| 항목 | 내용 |
|------|------|
| 소속 | 인디~중소 게임사 (팀 규모 5~30명) |
| 역할 | 게임 기획자 겸 LiveOps 운영 담당 |
| 기술 수준 | 비개발자. SQL 기초 이해 가능, 코드 작성 불가 |
| 주요 도구 | 스프레드시트, Notion, 간단한 BI 대시보드 |

**Goals**
- 코드 없이 이벤트를 직접 생성하고 시작/종료 시간을 설정하고 싶다.
- 주요 이벤트를 특정 플레이어 그룹(신규, VIP 등)에게만 노출하고 싶다.
- 이벤트 성과(참여율, 매출 영향)를 이벤트 종료 후 바로 확인하고 싶다.

**Pain Points**
- 이벤트 설정 변경마다 개발자를 거쳐야 해서 배포 사이클이 길다.
- 여러 이벤트가 동시에 진행될 때 일정 충돌을 캘린더에서 한눈에 확인하기 어렵다.
- 이벤트 기간 중 문제 발생 시 즉시 중단할 방법이 없다.

#### Secondary Persona: 박현수 (개발자/기술 운영자)

PRD-GLO-001에서 정의된 동일 페르소나를 재사용한다.

| 항목 | 내용 |
|------|------|
| 소속 | 중소 게임사 서버 개발자 |
| 역할 | LiveOps 백엔드 연동 및 고급 이벤트 관리 |
| 기술 수준 | SQL, 프로그래밍 언어 숙련 |

**Goals**
- SDK를 통해 현재 활성 이벤트 목록과 메타데이터를 조회하고 싶다.
- 이벤트 상태 변경 시 웹훅으로 자동 알림을 수신해 연동 시스템에 반영하고 싶다.

**Pain Points**
- 이벤트 배포 실수 발생 시 롤백 절차가 불명확하다.

#### Tertiary Persona: 이수정 (팀 리드/승인 권한자)

| 항목 | 내용 |
|------|------|
| 소속 | 인디~중소 게임사 LiveOps 팀 리드 |
| 역할 | 이벤트 검토 및 승인/반려 담당 |
| 기술 수준 | 비개발자. 이벤트 기획 경험 다수. |
| 주요 관심사 | 이벤트 배포 전 오류 방지, 팀 운영 품질 관리 |

**Goals**
- 승인 요청된 이벤트 내용을 한눈에 검토하고 빠르게 승인 또는 반려하고 싶다.
- 반려 시 사유를 기록해 기획자가 수정 방향을 명확히 알 수 있게 하고 싶다.
- 승인 없이 이벤트가 활성화되는 상황을 방지하고 싶다.

**Pain Points**
- 승인 요청이 왔는지 이메일/슬랙 외에 별도로 확인할 방법이 없다.
- 승인 시점의 이벤트 설정이 이후 수정되어도 알 수 없다.

### 2.2 사용자 시나리오

#### SC-004: 주말 이벤트 생성 및 승인 요청

> 김지연은 이번 주말 "주말 로그인 보상" 이벤트를 생성하고, 이수정에게 승인을 요청한다. 이수정은 이벤트 설정을 검토 후 승인하며, 이벤트는 설정된 시작 시간에 자동으로 활성화된다.

**Steps**
1. 김지연이 Live Events에서 새 이벤트 생성 → 이름, 이벤트 타입, 시작/종료 시간 입력
2. 오디언스 타겟팅 단계에서 "활성 플레이어" 오디언스 선택
3. metadata에 보상 정보(아이템 ID, 수량) 입력
4. "승인 요청" 버튼 클릭 → 이벤트 상태 pending_approval 전환
5. 이수정이 승인 요청 알림 수신 → 이벤트 상세 검토
6. 이수정이 "승인" 클릭 → 이벤트 상태 scheduled 전환
7. 토요일 오전 0시, 시스템이 start_at 도달 감지 → 이벤트 자동 active 전환
8. 일요일 오후 11시 59분, end_at 도달 → 이벤트 자동 ended 전환

#### SC-005: 반복 이벤트 스케줄링 (매주 금요일 출석 보상)

> 박현수는 매주 금요일 오전 9시에 자동으로 활성화되는 "금요일 출석 보상" 반복 이벤트를 설정한다. RFC 5545 rrule로 반복 규칙을 정의하며, 이벤트는 매주 24시간 동안 활성화된다.

**Steps**
1. 박현수가 새 이벤트 생성 → 이벤트 타입 "출석 보상" 선택
2. 스케줄링 단계에서 "반복 스케줄" 선택
3. rrule 입력: `FREQ=WEEKLY;BYDAY=FR`, 시작 시간 09:00, 지속 시간 24h 설정
4. 타임존 설정: `Asia/Seoul`
5. 승인 요청 → 승인 완료 → 매주 금요일 자동 활성화 확인

#### SC-006: 긴급 이벤트 일시정지 및 종료

> 김지연은 진행 중인 "봄맞이 이벤트"에서 보상 설정 오류를 발견한다. 즉시 이벤트를 일시정지하고, 팀과 협의 후 긴급 종료 처리한다.

**Steps**
1. 김지연이 활성 이벤트 목록에서 "봄맞이 이벤트" 확인
2. "일시정지(Pause)" 실행 → 사유 "보상 설정 오류" 입력 → 이벤트 상태 paused 전환, 플레이어에게 이벤트 노출 중단
3. 팀 내부 보상 오류 확인 및 협의
4. "긴급 종료(Kill)" 실행 → 종료 사유 입력 → 이벤트 상태 ended 전환
5. EVENT_STATE_LOG에 일시정지·종료 이력 기록 확인
6. 수정된 설정으로 새 이벤트 생성 후 재승인 요청

### 3.2 기능 상세: F-009 라이브 이벤트 CRUD

#### 개요

이벤트 기본 정보(이름, 설명, 이벤트 타입, 우선순위, 메타데이터)를 생성·조회·수정·삭제·복제한다. 유연한 콘텐츠 저장을 위해 하이브리드 콘텐츠 모델(구조화 필드 + `metadata` jsonb)을 채택한다.

#### 사용자 스토리

**US-017**
```
As a 게임 기획자(관리자),
I want to 이벤트 이름, 타입, 우선순위, 시작/종료 시간, 보상 메타데이터를 입력해 새 이벤트를 만들기를,
So that 코드 배포 없이 이벤트를 즉시 준비할 수 있다.
```

**US-018**
```
As a 게임 기획자(관리자),
I want to 기존 이벤트를 복제해 새 이벤트로 저장하기를,
So that 유사한 이벤트를 반복 생성할 때 처음부터 입력하지 않아도 된다.
```

**US-019**
```
As a 게임 기획자(관리자),
I want to draft 상태 이벤트를 삭제하고, 활성 이벤트는 archive 처리하기를,
So that 이벤트 목록을 정리하면서도 운영 이력을 보존할 수 있다.
```

#### 상세 요구사항

| ID | 요구사항 | 필수 |
|----|----------|------|
| REQ-009-01 | 이벤트는 name(필수), description(선택), event_type(필수), priority(필수), metadata(선택), payload_schema_version(선택)으로 구성된다. | Y |
| REQ-009-02 | name은 프로젝트 내에서 고유해야 하며, 최대 128자다. | Y |
| REQ-009-03 | event_type은 관리자가 자유롭게 정의하는 문자열 레이블이다 (예: `login_bonus`, `seasonal`, `tournament`). | Y |
| REQ-009-04 | priority는 `low`, `medium`, `high`, `critical` 4단계 중 하나를 선택한다. 동시 활성 이벤트 간 우선순위 표시에 사용된다. | Y |
| REQ-009-05 | metadata는 jsonb 타입으로 저장하며, 최대 64KB로 제한한다. 보상 정보, 이벤트 배너 URL, 게임 내 표시 데이터 등 비정형 콘텐츠를 저장한다. | Y |
| REQ-009-06 | payload_schema_version은 metadata 스키마 버전을 관리하는 문자열 필드다. SDK 클라이언트가 버전별로 다른 파싱 로직을 적용할 수 있도록 한다. | Y |
| REQ-009-07 | 이벤트 목록은 status, event_type, priority, 오디언스별로 필터링할 수 있어야 한다. | Y |
| REQ-009-08 | 이벤트 목록은 생성일, 시작 시간, 이름(가나다/ABC순) 기준으로 정렬할 수 있어야 한다. | Y |
| REQ-009-09 | 이벤트 복제 기능을 제공한다. 복제 시 status는 draft로, 스케줄(start_at, end_at)과 승인 정보는 초기화된다. 이름에 "(복사본)" 접미사가 자동 추가된다. | Y |
| REQ-009-10 | 삭제는 draft 상태에서만 가능하다. draft 외 상태의 이벤트는 archive 처리(ended → archived 전이)만 허용한다. | Y |
| REQ-009-11 | 이벤트 수정은 draft 상태에서만 자유롭게 가능하다. scheduled 이상의 상태에서 수정 시 재승인이 요구된다. | Y |

#### 수용 조건 (Acceptance Criteria)

**AC-009-01**
```gherkin
Scenario: 이벤트 신규 생성 성공
  Given 관리자가 Live Events 목록 페이지에 있다
  When 관리자가 name "주말 로그인 보상", event_type "login_bonus", priority "medium"을 입력하고 저장한다
  Then 이벤트가 status "draft"로 목록에 추가된다
  And 생성 시각이 기록된다
```

**AC-009-02**

### 3.3 기능 상세: F-010 이벤트 스케줄링

#### 개요

이벤트의 시작/종료 시간을 단순 지정하거나 RFC 5545 rrule 기반 반복 규칙으로 정의한다. 타임존을 IANA timezone DB 기준으로 지원하며, 내부 저장은 UTC로 통일한다. 다단계 페이즈 데이터 모델을 사전 설계하되 MVP는 단일 main phase만 사용한다.

#### 사용자 스토리

**US-020**
```
As a 게임 기획자(관리자),
I want to 이벤트 시작 시간과 종료 시간을 특정 날짜/시간으로 지정하기를,
So that 원하는 기간에만 이벤트가 자동으로 활성화되고 종료될 수 있다.
```

**US-021**
```
As a 개발자/기술 운영자(관리자),
I want to RFC 5545 rrule로 반복 스케줄을 설정하기를,
So that 매주 금요일, 매달 1일 등 주기적으로 반복되는 이벤트를 한 번 설정으로 자동 실행할 수 있다.
```

**US-022**
```
As a 게임 기획자(관리자),
I want to 이벤트 시간을 로컬 타임존(예: Asia/Seoul)으로 입력하기를,
So that 국내 플레이어 기준 시각을 UTC로 직접 환산하지 않아도 된다.
```

#### 상세 요구사항

| ID | 요구사항 | 필수 |
|----|----------|------|
| REQ-010-01 | 단순 스케줄: start_at(UTC), end_at(UTC)을 지정한다. 시작 시간은 종료 시간보다 이전이어야 한다. | Y |
| REQ-010-02 | display_timezone: IANA timezone DB 기준 타임존 문자열을 입력하면 관리자 화면에서 해당 타임존으로 시간을 표시한다. (예: `Asia/Seoul`, `America/Los_Angeles`) | Y |
| REQ-010-03 | 반복 스케줄: RFC 5545 rrule 문자열을 입력한다. (예: `FREQ=WEEKLY;BYDAY=FR`, `FREQ=MONTHLY;BYMONTHDAY=1`) | Y |
| REQ-010-04 | 반복 스케줄에서 각 반복 인스턴스의 지속 시간(duration)은 분 단위로 입력한다. | Y |
| REQ-010-05 | rrule 입력 시 다음 3회 예정 발동 시각을 미리보기로 표시한다. | Y |

| REQ-010-07 | 스케줄이 없는 이벤트(start_at NULL)는 draft 상태를 유지하며 승인 요청 시 start_at 필수 입력을 강제한다. | Y |

### 3.4 기능 상세: F-011 승인 워크플로우

**US-024**
```
As a 팀 리드/승인 권한자(관리자),
I want to 승인 요청된 이벤트를 검토하고 승인 또는 반려하기를,

```

**US-025**
```
As a 팀 리드/승인 권한자(관리자),
I want to 반려 시 사유를 입력하기를,
```

### 3.6 기능 상세: F-013 이벤트 상태 관리

#### 개요

이벤트의 생명 주기를 7단계 상태 머신으로 관리한다. 모든 상태 전이는 EVENT_STATE_LOG에 기록된다. 긴급 상황을 위한 Pause, Kill, Extend 제어 기능을 제공한다.

#### 사용자 스토리

**US-029**
```
As a 게임 기획자(관리자),
I want to 진행 중인 이벤트를 즉시 일시정지하기를,
So that 이벤트 설정 오류나 운영 이슈 발생 시 플레이어 피해를 최소화할 수 있다.
```

**US-030**
```
As a 게임 기획자(관리자),
I want to 이벤트 종료 시간을 연장하기를,
So that 이벤트 반응이 좋거나 긴급 이슈로 연장이 필요할 때 즉시 대응할 수 있다.
```

**US-031**
```
As a 게임 기획자(관리자),
I want to 이벤트의 모든 상태 전이 이력을 조회하기를,
So that 이벤트 운영 과정에서 발생한 변경 사항을 추적하고 감사할 수 있다.
```

#### 상태 정의

| 상태 | 설명 |
|------|------|
| `draft` | 초안. 생성 후 승인 요청 전 상태. 자유롭게 수정 가능. |
| `pending_approval` | 승인 대기 중. 수정 불가. |
| `scheduled` | 승인 완료. start_at 도달 대기 중. |
| `active` | 이벤트 활성화. 플레이어에게 노출 중. |
| `paused` | 일시정지. 플레이어에게 노출 중단. |
| `ended` | 종료됨 (자동 또는 수동). |
| `archived` | 보관 처리됨. 읽기 전용. |

#### 상태 전이 규칙

| 전이 | 트리거 | 조건 |
|------|--------|------|
| draft → pending_approval | 승인 요청 (수동) | 필수 필드 완성 |
| pending_approval → scheduled | 승인 완료 (수동, event_approver) | - |
| pending_approval → draft | 반려 (수동, event_approver) | 반려 사유 입력 필수 |
| scheduled → active | start_at 도달 (시스템 자동) | - |
| active → paused | Pause 실행 (수동) | 사유 입력 필수 |
| paused → active | 재개 실행 (수동) | - |
| active → ended | end_at 도달 (시스템 자동) 또는 Kill 실행 (수동) | Kill 시 사유 입력 필수 |
| paused → ended | Kill 실행 (수동) | 사유 입력 필수 |
| ended → archived | 보관 처리 (수동) | - |

#### 상세 요구사항

| ID | 요구사항 | 필수 |
|----|----------|------|
| REQ-013-01 | 모든 상태 전이는 EVENT_STATE_LOG에 기록된다: event_id, from_status, to_status, actor_id(시스템 자동 전이는 NULL), reason, created_at. | Y |
| REQ-013-02 | Pause(일시정지): active 상태에서만 가능. 즉시 플레이어 노출 중단. 자동 재개 타이머를 제공하지 않는다(명시적 재개 원칙). 사유 입력 필수. | Y |
| REQ-013-03 | Kill(긴급 종료): active 또는 paused 상태에서 가능. 이벤트를 즉시 ended 상태로 전환. 사유 입력 필수. | Y |
| REQ-013-04 | Extend(기간 연장): active 상태에서만 가능. end_at을 현재 설정보다 이후 시간으로 변경. 재승인 불필요(긴급 운영 대응). | Y |
| REQ-013-05 | Extend는 승인 없이 즉시 적용되나, EVENT_STATE_LOG에 연장 전후 end_at과 실행자가 기록된다. | Y |
| REQ-013-06 | start_at 도달 시 시스템이 scheduled → active 자동 전이를 처리한다. 전이 처리 지연은 1초 이내여야 한다. | Y |
| REQ-013-07 | end_at 도달 시 시스템이 active → ended 자동 전이를 처리한다. | Y |
| REQ-013-08 | EVENT_STATE_LOG 데이터 보존 기간은 1년이다. 1년 초과 로그는 자동 삭제된다. | Y |

**AC-013-01**

**AC-013-04**
```gherkin

  Given "주말 이벤트"가 active 상태이고 end_at이 "2026-04-06 23:59 UTC"이다

  Then end_at이 즉시 갱신된다
  And 재승인 없이 변경이 적용된다

```

```gherkin

  When 관리자가 사유를 입력하지 않고 Pause를 실행하려 한다
  Then Pause가 차단된다
  And "일시정지 사유를 입력해야 합니다." 오류 메시지가 표시된다
```

| ID | 규칙 | 예시 |

### 4.1 성능

| 항목 | 요구사항 | 측정 기준 |
|------|----------|----------|
| 동시 활성 이벤트 수 | 프로젝트당 50개 이상 정상 지원 | 동시 active 이벤트 50개 상태에서 SDK p99 응답 측정 |
| SDK 이벤트 조회 응답 시간 | p99 100ms 이하 | 플레이어 SDK의 활성 이벤트 목록 조회 API 응답 시간 |
| SDK 응답 캐시 TTL | 1분 | 이벤트 상태 변경 ~ SDK 캐시 반영 최대 지연 |
| 상태 전이 자동 처리 지연 | 1초 이내 | start_at/end_at 도달 ~ 상태 전이 완료 시간 |
| 관리자 화면 초기 로딩 | 95th percentile 3초 이내 | 페이지 첫 요청 ~ 인터랙션 가능 상태 |
| 성과 대시보드 집계 지연 | 최대 1시간 | 이벤트 데이터 발생 ~ EVENT_METRICS 반영 시간 |

### 4.2 보안

| 항목 | 요구사항 |
|------|----------|
| 인증 | 관리자 대시보드: 세션 기반 인증. SDK/REST API: API Key 기반 인증. |
| 권한 | 이벤트 CRUD: 프로젝트 편집 권한. 승인/반려: `event_approver` 역할. 조회: 프로젝트 읽기 권한. |
| 데이터 접근 격리 | 프로젝트(게임) 간 이벤트 데이터 완전 격리. 타 프로젝트 이벤트 참조 불가. |
| 웹훅 보안 | 웹훅 시크릿을 이용한 HMAC-SHA256 서명 검증. 시크릿은 마스킹 처리 후 표시. |
| 감사 로그 | 이벤트 생성·수정·승인·긴급 제어 등 주요 관리 행위를 EVENT_STATE_LOG 및 EVENT_APPROVAL에 기록. |

### 4.3 확장성

- 이벤트 자동 전이 스케줄러는 프로젝트 수 및 예약된 이벤트 수 증가에 따라 수평 확장 가능하도록 설계한다.
- SDK 응답 캐시 레이어는 동시 활성 이벤트 증가(50개 → 200개)에도 p99 100ms 기준을 유지해야 한다.
- EVENT_STATE_LOG와 EVENT_APPROVAL는 파티션 기반으로 관리해 1년 보존 기간 내 쿼리 성능을 보장한다.

### 4.4 가용성

| 항목 | 요구사항 |
|------|----------|
| SLA | 월간 가용성 99.9% (다운타임 월 43분 이내) |
| RTO (복구 목표 시간) | 1시간 |
| RPO (복구 목표 시점) | 1시간 |
| 자동 전이 장애 시 | 스케줄러 장애 복구 후 미처리된 전이를 즉시 일괄 처리. 과거 start_at 도달 이벤트는 복구 즉시 active 전환. |

### 4.5 데이터 보존

| 데이터 | 보존 기간 |
|--------|----------|
| EVENT_STATE_LOG | 1년 (이후 자동 삭제) |
| EVENT_APPROVAL | 1년 (이후 자동 삭제) |

### 5.2 제약사항

**기술적 제약**
- 프론트엔드 기술 스택은 Next.js 16, React 19, TypeScript, Shadcn UI, Radix UI, Tailwind CSS 4로 고정한다. (CLAUDE.md)
- 차트 시각화는 Recharts 라이브러리를 사용한다.
- metadata의 jsonb 스키마 유효성 검사는 시스템이 수행하지 않는다. 게임 클라이언트(SDK) 측 책임이다.

**일정 제약**
- Phase 1 (MVP) 출시 전에 PRD-GLO-001이 완료되어야 한다 (AUDIENCE 테이블 의존성).
- Phase 1 출시 대상: F-009 ~ F-016 전체.

**리소스 제약**
- 게임 경제 시스템(상점, 가상 화폐, 보상 정산) 미구현 상태이므로 보상/경제 데이터는 `metadata` jsonb에 게임 클라이언트가 해석하는 방식으로 저장한다. GLO 시스템이 직접 보상을 지급하지 않는다.

### 5.3 가정사항

- 플레이어의 이벤트 참여 행동(참여, 완료) 데이터는 SDK 이벤트를 통해 GLO로 전달된다고 가정한다.
- 보상 정산 로직은 게임 서버가 `metadata`를 파싱하여 직접 처리한다고 가정한다. GLO는 이벤트 조건과 메타데이터만 제공한다.
- 반복 이벤트 인스턴스의 개별 수정은 Phase 2에서 검토한다.

## Appendix

### A. 용어 정의

| 용어                     | 정의                                                                                                |
| ---------------------- | ------------------------------------------------------------------------------------------------- |
| 라이브 이벤트 (Live Event)   | 특정 기간 동안 활성화되어 플레이어에게 노출되는 게임 내 이벤트. 보상, 콘텐츠 해금, 특별 모드 등을 포함.                                     |
| 이벤트 상태 머신              | 이벤트가 거치는 7단계(draft → pending_approval → scheduled → active → paused → ended → archived) 상태 전이 체계. |
| version_snapshot       | 이벤트 승인 시점의 전체 설정 데이터를 JSON으로 저장한 불변 스냅샷. 승인 후 변경 감지에 사용.                                          |
| sticky_membership      | 이벤트 활성화 시점의 오디언스 멤버를 고정하는 옵션. 이후 오디언스 조건 이탈과 무관하게 이벤트 참여 자격 유지.                                   |
| rrule                  | RFC 5545 표준에 따른 반복 규칙 문자열. 이벤트의 주기적 발동 스케줄 정의에 사용. (예: `FREQ=WEEKLY;BYDAY=FR`)                    |
| payload_schema_version | metadata jsonb의 스키마 버전을 식별하는 문자열. SDK 클라이언트가 버전별 파싱 로직을 적용하는 데 사용.                                |
| 긴급 제어                  | 운영 중 이벤트에 즉시 적용되는 조치: Pause(일시정지), Kill(긴급 종료), Extend(기간 연장).                                    |
| EVENT_STATE_LOG        | 이벤트의 모든 상태 전이 이력을 기록하는 append-only 로그 테이블.                                                        |
| EVENT_APPROVAL         | 이벤트 승인 워크플로우의 요청·처리 이력과 version_snapshot을 저장하는 테이블.                                               |
| event_approver         | 이벤트 승인 및 반려 권한을 가진 RBAC 역할.                                                                       |
| EVENT_PHASE            | 이벤트의 다단계 페이즈(main, warmup, cooldown, bonus)를 정의하는 테이블. MVP에서는 main phase만 사용.                     |
| EVENT_METRICS          | 이벤트 성과 지표(참여, 수익)를 집계 저장하는 테이블. 최대 1시간 지연 배치 집계.                                                  |

### B. 하이브리드 콘텐츠 모델 상세

metadata jsonb 스키마 예시 (payload_schema_version: "v1"):

```json
{
  "banner_url": "https://cdn.example.com/events/spring2026/banner.png",
  "rewards": [
    {
      "item_id": "item_gold_coin",
      "quantity": 100,
      "condition": "login"
    },
    {
      "item_id": "item_special_chest",
      "quantity": 1,
      "condition": "purchase"
    }
  ],
  "display": {
    "title_i18n": {
      "ko": "봄맞이 이벤트",
      "en": "Spring Event"
    },
    "color_theme": "#FF6B6B"
  },
  "game_config": {
    "exp_multiplier": 2.0,
    "drop_rate_boost": 1.5
  }
}
```

**설계 원칙**:
- GLO는 metadata 내용을 해석하거나 검증하지 않는다.
- payload_schema_version으로 SDK가 버전별 파싱 로직을 분기한다.
- 보상 지급 로직은 게임 서버 책임이다.
