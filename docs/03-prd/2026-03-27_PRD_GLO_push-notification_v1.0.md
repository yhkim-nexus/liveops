## 1. Overview

### 1.1 제품 비전

운영팀이 개발자 없이 관리 콘솔에서 캠페인성 모바일 푸시 알림을 직접 작성·타겟팅·스케줄링·발송하고, 발송 결과를 분석할 수 있는 엔드투엔드 캠페인 관리 시스템을 구축한다. FCM/APNs 기반의 안정적인 발송 인프라 위에서 메시지 작성, 오디언스 타겟팅, 빈도 제한, A/B 테스트까지 푸시 알림 운영 전체 라이프사이클을 지원한다.

### 1.2 배경 및 목적

**배경**

- SES-GLO-001 킥오프에서 "푸시 알림 / 인앱 메시징"이 기획 범위에 포함되었다.
- 경쟁사 분석(RES-GLO-002)에서 5/7 플랫폼이 푸시 알림 발송 기능을 지원하며, One Signal과 Braze가 캠페인 관리, A/B 테스트, 발송 분석 기능을 필수로 제공한다.
- PRD-GLO-002 F-016에서 이벤트 기반 내부 알림(웹훅)을 정의했으나, 운영팀이 직접 캠페인성 메시지를 발송하는 기능은 별도 확장이 필요하다.
- PRD-GLO-001 F-006 세그멘테이션 시스템의 오디언스 타겟팅 인프라를 재사용하여 정교한 발송 대상 설정이 가능하다.
- PRD-GLO-003 A/B 테스트 실험 프레임워크와 연동하여 메시지 최적화를 지원한다.

**목적**

- 이 PRD는 GLO 서비스의 **모바일 푸시 알림 발송** 기능 범위를 정의한다.
- 메시지 작성(F-037), 타겟 오디언스 설정(F-038), 발송 스케줄링(F-039), 템플릿 관리(F-040), 발송 분석(F-041), 빈도 제한(F-042), 푸시 A/B 테스트(F-043)를 명세한다.
- MVP(F-037, F-038, F-039, F-042) 출시에 필요한 최소 기능과 수용 기준을 확정한다.

## 2. Target Users

### 2.1 사용자 페르소나

#### Primary Persona: 김민수 (LiveOps 운영자 / Editor)

| 항목 | 내용 |
|------|------|
| 소속 | 게임사 LiveOps 운영팀 |
| 역할 | 푸시 캠페인 기획 및 초안 작성, 오디언스 타겟팅, 스케줄링 설정 |
| 기술 수준 | 비개발자. 마케팅 툴 사용 경험 보유 |
| 주요 도구 | 관리자 대시보드, 스프레드시트, 메신저 |

**Goals**
- 코드 작성 없이 캠페인 메시지를 빠르게 작성하고 예약 발송을 설정하고 싶다.
- 활성 회원 중 특정 레벨대나 오디언스에만 타겟팅하여 불필요한 알림을 줄이고 싶다.
- 지난 캠페인 오픈율과 전환율을 확인하여 다음 캠페인 메시지를 개선하고 싶다.

**Pain Points**
- 발송 대상을 설정할 때 개발자에게 요청해야 하는 병목이 있다.
- 예약 발송 후 실제로 발송되었는지, 도달했는지 확인할 방법이 없다.
- 잘 작동했던 메시지를 매번 새로 작성해야 하는 비효율이 있다.

#### Secondary Persona: 이수정 (팀 리드 / Operator)

| 항목 | 내용 |
|------|------|
| 소속 | 게임사 운영팀 리드 |
| 역할 | 대량 발송 캠페인 최종 승인, 오발송 방지 검토 |
| 기술 수준 | 비개발자. 데이터 분석 기초 이해 |

**Goals**
- 대량 발송 전에 메시지 내용과 타겟 오디언스를 최종 검토하고 승인하고 싶다.
- 전체 캠페인 성과 지표를 한눈에 파악하고 싶다.

**Pain Points**
- 오발송이 발생했을 때 즉시 취소할 방법이 없다.
- 발송 승인을 받기 위해 카카오톡으로 메시지 내용을 공유하는 비효율이 있다.

#### Tertiary Persona: 박지연 (개발자 / Admin)

| 항목 | 내용 |
|------|------|
| 소속 | 게임사 클라이언트/서버 개발팀 |
| 역할 | FCM/APNs SDK 연동, 푸시 토큰 등록/갱신, 딥링크 처리 |
| 기술 수준 | 개발자. REST API, SDK 연동 경험 풍부 |

**Goals**
- FCM/APNs 인증 키를 Settings 페이지에서 안전하게 등록하고 싶다.
- 잘못된 토큰을 자동으로 정리하여 불필요한 발송 비용을 줄이고 싶다.

### 2.2 사용자 시나리오

#### SC-010: 즉시 발송 — 긴급 점검 공지

> 이수정(팀 리드)은 예상치 못한 서버 점검이 발생했을 때, 전체 회원에게 긴급 점검 알림을 방해 금지 시간을 무시하고 즉시 발송해야 한다.

**Steps**
1. 관리 콘솔에서 "푸시 캠페인 생성" 버튼을 클릭한다.
2. 메시지 작성 단계에서 제목 "긴급 점검 안내"와 본문을 입력하고 Priority를 "critical"로 설정한다.
3. 오디언스 설정 단계에서 "전체 회원"을 선택하고 예상 도달 수를 확인한다.
4. 스케줄링 단계에서 "즉시 발송"을 선택한다.
5. 확인 단계에서 내용을 검토한 후 "발송 승인" 버튼을 클릭한다.
6. 발송이 시작되고 전송 현황이 실시간으로 업데이트된다.

#### SC-011: 예약 반복 발송 — 주간 이벤트 알림

> 김민수(운영자)는 매주 월요일 오전 10시에 활성 회원 중 레벨 10 이상인 오디언스에게 주간 이벤트 알림을 반복 발송하고 싶다.

**Steps**
1. 푸시 캠페인 생성 위저드를 시작한다.
2. 메시지 작성 단계에서 주간 이벤트 안내 메시지를 작성하고 iOS/Android 미리보기를 확인한다.
3. 오디언스 설정 단계에서 "레벨 10 이상 활성 회원" 오디언스를 선택한다.
4. 스케줄링 단계에서 "반복 발송"을 선택하고 "매주 월요일 10:00 KST"를 설정한다.
5. 확인 단계에서 승인 요청을 제출한다.
6. 이수정(팀 리드)이 승인 후, 매주 월요일마다 자동으로 발송된다.

#### SC-012: A/B 테스트 — 오픈율 최적화

> 김민수(운영자)는 복귀 유도 메시지 두 가지 변형을 테스트하여 오픈율이 높은 버전을 확인하고 이후 캠페인에 활용한다.

**Steps**
1. 푸시 A/B 테스트 캠페인 생성을 시작한다.
2. 변형 A("오늘의 보상을 받아가세요!")와 변형 B("3일 접속 보상이 기다려요!")를 각각 작성한다.
3. 트래픽 분배를 50:50으로 설정한다.
4. 7일 후 오픈율 기준으로 자동 승자 선정 옵션을 활성화한다.
5. 동일한 타겟 오디언스와 스케줄을 설정하고 발송 승인을 요청한다.
6. 7일 후 오픈율 기반으로 변형 B가 승자로 자동 선정된다.

### 3.2 F-037: 푸시 메시지 작성

**설명:** 운영자가 캠페인 생성 위저드의 첫 번째 단계에서 푸시 알림 메시지를 작성하고 iOS/Android 디바이스 프레임에서 미리보기를 확인한다.

#### 유저 스토리

| ID | 스토리 | 우선순위 |
|----|--------|---------|
| US-056 | 운영자로서, 제목과 본문을 입력하면 iOS/Android 디바이스 화면에서 알림이 어떻게 보이는지 즉시 확인할 수 있다 | P0 |
| US-057 | 운영자로서, 알림에 딥링크 URL을 설정하여 클릭 시 특정 게임 화면으로 이동하도록 설정할 수 있다 | P0 |
| US-058 | 운영자로서, 이미지 URL을 추가하여 리치 푸시 알림을 발송할 수 있다 | P0 |
| US-059 | 운영자로서, 커스텀 JSON 데이터를 페이로드에 포함하여 클라이언트 앱에서 추가 처리를 할 수 있도록 설정할 수 있다 | P1 |

| REQ-037-06 | 제목과 본문 입력 시 iOS 및 Android 디바이스 프레임 미리보기를 실시간으로 업데이트한다 | P0 |
| REQ-037-07 | iOS 미리보기와 Android 미리보기를 탭 전환 방식으로 제공한다 | P0 |

| REQ-037-09 | 다국어 메시지 지원(언어별 제목/본문 입력)은 Phase 2에서 구현한다 (Out of Scope) | - |

#### 수용 기준 (Acceptance Criteria)

**AC-037-01: 제목 입력 및 자수 카운터**

```gherkin
Scenario: 제목 입력 시 자수 카운터 실시간 업데이트
  Given 캠페인 생성 위저드 1단계(메시지 작성)가 열려 있다
  When 제목 필드에 "오늘의 이벤트를 확인하세요!" 를 입력한다 (14자)
  Then 제목 필드 하단에 "14 / 50" 자수 카운터가 표시된다
  And 미리보기 패널의 제목 영역이 즉시 업데이트된다
```

**AC-037-02: 제목 최대 자수 초과 방지**

```gherkin
Scenario: 제목 50자 초과 입력 방지
  Given 제목 필드에 이미 50자가 입력된 상태이다
  When 추가 문자를 입력하려 한다
  Then 입력이 차단되고 카운터가 "50 / 50"으로 표시된다
```

**AC-037-03: 이미지 URL 유효성 검사**

```gherkin
Scenario: 유효하지 않은 이미지 URL 입력 시 오류 표시
  Given 이미지 URL 필드가 활성화된 상태이다
  When ".gif" 확장자 URL을 입력한다
  Then "JPG 또는 PNG 형식의 이미지 URL만 허용됩니다" 오류 메시지가 표시된다
  And 다음 단계로 진행하는 버튼이 비활성화된다
```

**AC-037-04: 커스텀 데이터 JSON 유효성 검사**

```gherkin
Scenario: 잘못된 JSON 형식 입력 시 오류 표시
  Given 커스텀 데이터 필드가 활성화된 상태이다
  When "{ key: value }" (따옴표 누락) 를 입력한다
```

| US-067 | 팀 리드로서, 대량 발송 캠페인을 최종 검토하고 승인 또는 반려할 수 있다                         | P0   |

### 3.7 F-042: 빈도 제한 및 발송 보호

**설명:** 플레이어 경험을 보호하기 위해 플레이어당 일일 최대 푸시 발송 수를 제한한다. 야간 방해 금지 처리는 서버에서 관리하지 않고 **플레이어 디바이스의 OS 방해 금지 모드(iOS Focus / Android DND)에 위임**한다. 이를 통해 MVP 서버 복잡도를 최소화하면서 글로벌 환경에서도 플레이어 경험을 보호한다.

> **설계 결정 (D-041):** 방해 금지 시간을 서버 사이드에서 관리하지 않고 OS에 위임한다.
> - iOS의 Focus 모드, Android의 DND(Do Not Disturb) 모드가 이미 야간 알림 제어를 제공한다.
> - 글로벌 서비스에서 국가별 야간 시간대 규제(한국 21:00, EU 22:00 등)를 서버가 관리하면 복잡도가 크게 증가한다.
> - 대부분의 경쟁 플랫폼(Unity, GameAnalytics)도 MVP에서는 OS 위임 방식을 사용한다.
> - v1.1에서 서버 사이드 방해 금지(지연 발송 큐, 국가별 기본값)를 선택적으로 추가할 수 있다.

```gherkin
  Given 플레이어 A의 일일 최대 발송 수가 3회로 설정되어 있다
```

**AC-042-03: Critical Priority 선택 시 확인 다이얼로그**

**AC-042-04: 야간 시간 발송은 OS에 위임됨**

```gherkin
  Given 플레이어 B의 로컬 시간이 새벽 3시이다
  And 플레이어 B의 디바이스에 DND 모드가 설정되어 있다
```

### 3.8 F-043: 푸시 A/B 테스트

**설명:** 최대 4개의 메시지 변형을 생성하고 트래픽을 분배하여 어떤 메시지가 더 높은 오픈율 또는 전환율을 달성하는지 테스트한다. PRD-GLO-003 실험 프레임워크와 연동한다.

#### 유저 스토리

| ID | 스토리 | 우선순위 |
|----|--------|---------|
| US-078 | 운영자로서, 동일한 오디언스에 서로 다른 메시지 변형을 발송하여 어떤 메시지가 더 효과적인지 비교할 수 있다 | P2 |
| US-079 | 운영자로서, A/B 테스트 종료 후 오픈율 기반으로 승자가 자동으로 선정되어 나머지 대상에게 자동 발송되도록 설정할 수 있다 | P2 |
| US-080 | 운영자로서, A/B 테스트 결과를 변형별 오픈율·전환율·통계적 유의성과 함께 확인할 수 있다 | P2 |

#### 요구사항

| ID | 요구사항 | 우선순위 |

| REQ-043-03 | 자동 승자 선정 기능을 제공한다: 설정한 기간(1/3/7일) 이후 오픈율 또는 전환율 기준으로 승자 변형을 자동 선정하고 나머지 대상에게 자동 발송한다 | P2 |

| REQ-043-06 | 최소 2개 이상의 변형이 설정되어야 A/B 테스트 캠페인을 승인 요청할 수 있다 | P2 |

#### 수용 기준 (Acceptance Criteria)

**AC-043-01: 변형 생성 및 트래픽 분배**

```gherkin
Scenario: 2개 변형 A/B 테스트 설정
  Given A/B 테스트 캠페인 생성 위저드가 열려 있다
  When 변형 A("오늘의 보상을 받아가세요!"), 변형 B("3일 접속 보상이 기다려요!")를 작성하고 트래픽을 50:50으로 설정한다
  Then 변형 A에 전체 오디언스의 50%가, 변형 B에 50%가 할당된다
```

**AC-043-02: 자동 승자 선정**

```gherkin
Scenario: 7일 후 오픈율 기반 자동 승자 선정
  Given A/B 테스트가 7일 기간으로 오픈율 기준 자동 승자 선정 옵션이 활성화된 상태이다
  When 7일이 경과하고 변형 B의 오픈율(8.2%)이 변형 A(5.1%)보다 높다
  Then 변형 B가 승자로 자동 선정된다
  And 나머지 미발송 대상에게 변형 B 메시지가 자동 발송된다
```

**AC-043-03: 트래픽 분배 합산 검증**

```gherkin
Scenario: 트래픽 합계가 100%를 초과하면 저장 불가
  Given 3개 변형이 생성된 상태이다
  When 각 변형의 트래픽을 40%, 40%, 30%로 설정한다
  Then "트래픽 분배의 합계가 100%여야 합니다 (현재: 110%)" 오류 메시지가 표시된다
  And 승인 요청 버튼이 비활성화된다
```

#### 비즈니스 규칙

| ID | 규칙 | 예시 |
|----|------|------|
| BR-043-01 | 변형 수는 최소 2개, 최대 4개이다 | 1개 또는 5개 설정 불가 |
| BR-043-02 | 트래픽 분배 합계는 반드시 100%여야 한다 | 50% + 30% = 오류 |
| BR-043-03 | 자동 승자 선정 기간은 최소 1일, 최대 30일이다 | 31일 설정 불가 |
| BR-043-04 | 통계적 유의성은 95% 신뢰 수준(p < 0.05)을 기준으로 표시한다 | |
| BR-043-05 | A/B 테스트 진행 중에는 변형 내용이나 트래픽 비율을 수정할 수 없다 | 발송 시작 후 수정 불가 |

| 상황 | 처리 | 메시지 | 코드 |

## 5. Data Model

### 5.1 PUSH_CAMPAIGN 엔티티

```typescript
type CampaignStatus =
  | "draft"
  | "pending_approval"
  | "approved"
  | "scheduled"
  | "sending"
  | "sent"
  | "completed"
  | "failed"
  | "cancelled";

type ScheduleType = "immediate" | "scheduled" | "recurring";
type Priority = "normal" | "critical";
type AudienceType = "all" | "audience" | "individual";

interface PushCampaign {
  id: string;                     // 캠페인 고유 ID
  name: string;                   // 캠페인 이름
  status: CampaignStatus;         // 캠페인 상태
  priority: Priority;             // 발송 우선순위 (normal / critical)

  // 메시지 (F-037)
  title: string;                  // 알림 제목 (최대 50자)
  body: string;                   // 알림 본문 (최대 200자)
  deepLinkUrl?: string;           // 딥링크 URL (선택)
  imageUrl?: string;              // 이미지 URL (선택)
  customData?: Record<string, unknown>; // 커스텀 데이터 페이로드 (최대 4KB JSON)

  // 오디언스 (F-038)
  audienceType: AudienceType;
  audienceIds?: string[];         // 선택된 오디언스 ID 목록 (audience 타입 시)
  playerIds?: string[];           // 개별 회원 ID 목록 (individual 타입 시)
  pushOptInFilter: boolean;       // 수신 동의 필터 여부 (기본 true)
  platformFilter: "all" | "ios" | "android";

  // 스케줄링 (F-039)
  scheduleType: ScheduleType;
  scheduledAt?: string;           // 예약 발송 일시 (ISO 8601)
  timezone?: string;              // 타임존 (IANA timezone, 예: "Asia/Seoul")
  rrule?: string;                 // RFC 5545 반복 규칙 (반복 발송 시)
  usePlayerTimezone: boolean;     // 플레이어 로컬 타임존 기반 발송 여부

  // 템플릿 연동 (F-040)
  templateId?: string;            // 사용된 템플릿 ID (선택)

  // A/B 테스트 연동 (F-043)
  isAbTest: boolean;
  abExperimentId?: string;        // PRD-GLO-003 실험 ID 연동

  // 승인 워크플로우
  createdBy: string;              // 작성 관리자 ID
  approvedBy?: string;            // 승인 관리자 ID
  rejectedBy?: string;            // 반려 관리자 ID
  rejectionReason?: string;       // 반려 사유

  // 메타
  estimatedReach: number;         // 예상 도달 수
  createdAt: string;              // 생성 일시 (ISO 8601)
  updatedAt: string;              // 수정 일시 (ISO 8601)
  sentAt?: string;                // 발송 시작 일시
  completedAt?: string;           // 발송 완료 일시
}
```

## 7. Dependencies & Constraints

### 7.1 선행 의존성

| 의존 대상 | 문서 ID | 의존 유형 | 설명 |
|-----------|---------|----------|------|
| 오디언스 시스템 | PRD-GLO-001 (F-006) | 하드 의존성 | F-038 타겟 오디언스 설정에서 오디언스 목록과 멤버십 데이터를 직접 사용한다 |
| RBAC 시스템 | PRD-GLO-005 | 하드 의존성 | Editor 역할은 캠페인 초안 작성, Operator 이상 역할만 발송 승인 가능 |
| 2단계 승인 패턴 | PRD-GLO-002 | 패턴 재사용 | F-039 발송 승인 워크플로우는 PRD-GLO-002 라이브 이벤트의 승인 패턴을 따른다 |
| A/B 테스트 프레임워크 | PRD-GLO-003 | 소프트 의존성 | F-043 푸시 A/B 테스트 결과를 PRD-GLO-003 실험 프레임워크에 연동한다. F-043은 v1.2 스코프이므로 MVP에는 필요 없다 |

### 7.2 외부 의존성

| 시스템 | 설명 | 의존 기능 |
|--------|------|----------|
| FCM (Firebase Cloud Messaging) | Android 푸시 발송 | F-037~F-043 전체 |
| APNs (Apple Push Notification service) | iOS 푸시 발송 | F-037~F-043 전체 |
| Job Queue (Redis/SQS) | 스케줄링, 대량 발송 배치 처리 | F-039, F-042 |

### 7.3 제약사항

| 제약 | 설명 |
|------|------|
| FCM/APNs 인증 키 미등록 | FCM/APNs 인증 키가 Settings에 등록되지 않으면 발송 기능 전체가 비활성화된다 |
| 수신 동의 데이터 | 수신 동의 여부는 클라이언트 SDK에서 수집하여 서버에 저장된다. 관리 콘솔은 조회와 필터링만 수행한다 (D-037) |
| Mock 데이터 전용 | 현재 구현은 Mock API 기반이며 실제 FCM/APNs 연동은 백엔드 구현 후 반영된다 |
| 리치 푸시 MVP 범위 | MVP에서는 이미지만 지원한다. 커스텀 액션 버튼은 Phase 2 스코프이다 (D-038) |

### 7.4 가정사항

- 게임 클라이언트 SDK가 FCM/APNs 푸시 토큰을 수집하여 서버에 등록한다.
- 수신 동의(opt-in) 상태가 플레이어 레코드에 저장되어 있다.
- 플레이어의 로컬 타임존 정보가 플레이어 레코드에 저장되어 있다.
- Job Queue 인프라(Redis 또는 AWS SQS)가 백엔드에서 제공된다.

### A. 용어 정의

| 용어 | 정의 |
|------|------|
| 캠페인 | 특정 오디언스에게 특정 시간에 발송하는 푸시 알림 발송 단위 |
| 오디언스 | PRD-GLO-001에서 정의된 세그멘테이션 기반의 발송 대상 그룹 |
| 예상 도달 수 (Estimated Reach) | 오디언스 크기 × 수신 동의율 × 플랫폼 비율로 계산된 추정 발송 가능 인원 수 |
| 빈도 제한 (Frequency Cap) | 플레이어당 일일 최대 발송 수 제한 설정 |
| 방해 금지 시간 (Quiet Hours) | 플레이어 로컬 타임존 기준 알림 발송이 차단되는 야간 시간대 |
| 딥링크 (Deep Link) | 알림 클릭 시 게임 앱의 특정 화면으로 직접 이동하는 URL |
| rrule | RFC 5545에서 정의한 반복 이벤트 규칙 문자열 형식 |
| FCM | Firebase Cloud Messaging. Google의 Android 및 iOS 푸시 발송 서비스 |
| APNs | Apple Push Notification service. Apple의 iOS 푸시 발송 서비스 |
| 수신 동의 (Opt-in) | 플레이어가 모바일 푸시 알림 수신에 동의한 상태 |
